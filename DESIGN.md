## Problem Statement

autoxmimsim should take one or more measured spectra and automatically find the optimized geometry and composition of the measured sample. The user may provide starting values, but the tool should do the rest.

The first milestone should start with synthetic spectra instead of measured spectra. Synthetic targets give known ground truth, so the project can prove that the inverse workflow recovers the original geometry and composition before handling real experimental noise and calibration issues.

## What Makes This Cool

The cool version is not another peak fitting tool. It is an inverse XMI-MSIM workflow:

1. Generate or load a target spectrum.
2. Run XMI-MSIM candidate simulations with varied geometry and composition.
3. Compare whole spectra directly.
4. Optimize toward the target.
5. Report the best parameters, uncertainty estimates, residuals, and evidence that the result is trustworthy.

The eventual 10x version adds full scientific reports, Bayesian active-learning optimization, and a GUI where the user can watch candidate spectra converge toward the target.

## Constraints

- The project uses the existing Python 3.10 Conda environment named `py10`.
- The first backend should use the Python wrapper for XMI-MSIM if practical.
- Speed matters because Monte Carlo simulations are expensive.
- Accuracy and uncertainty estimates matter more than UI polish in the first milestone.
- The initial target should be synthetic so success can be evaluated against known truth.
- The design should not require classical peak fitting as the central method. Direct spectrum comparison is the initial core.

## Premises

1. The first milestone should use synthetic target spectra so geometry and composition recovery can be measured objectively.
2. The first objective should compare whole spectra directly rather than beginning with peak fitting.
3. `xmimsim` should be the first backend adapter, but autoxmimsim should hide it behind its own interface so the backend can change later.
4. Bayesian optimization is the right long-term optimizer, but the first implementation should include a simple baseline optimizer to prove the pipeline and compare performance.
5. Reports should be part of the project identity, but the first report can be simple: truth, recovered parameters, error, fit plot, residual plot, and uncertainty summary.

## Landscape Notes

- XMI-MSIM is the right physical forward model: an open-source Monte Carlo simulator for energy-dispersive XRF spectra.
- The `xmimsim` Python package is a plausible first control layer, although it is old and should be isolated behind an adapter.
- Existing XRF tools such as PyMca and Peakaboo focus more on peak fitting and analysis workflows.
- Bayesian optimization libraries are mature, but some current packages have Python version constraints. Since this project is pinned to Python 3.10, optimizer dependency choices need to be checked carefully.

## Approaches Considered

### Approach A: Minimal Synthetic Recovery

Build a CLI/API that creates one synthetic target spectrum, searches over a small parameter space, compares spectra directly, and reports recovered parameters.

Effort: S
Risk: Low

Pros:

- Fastest useful proof.
- Easy to test.
- Keeps the first milestone narrow.

Cons:

- No GUI.
- Uncertainty may be simple at first.
- Limited parameter space.

Reuses:

- Current package scaffold.
- `xmimsim`.
- NumPy/SciPy or another lightweight optimizer.

### Approach B: Scientific Core Architecture

Build proper internal layers now: spectrum representation, parameter spaces, simulation backend adapter, objective functions, optimizer interface, uncertainty module, and report generator.

Effort: M/L
Risk: Medium

Pros:

- Best foundation for accuracy, uncertainty, speed comparisons, and future GUI.
- Keeps XMI-MSIM integration swappable.
- Makes synthetic and measured workflows share the same core.

Cons:

- More upfront design.
- Slower first visible result than a script.
- Requires discipline to keep milestone one small.

Reuses:

- `xmimsim` as the first simulation backend.
- NumPy/SciPy for spectra, metrics, and a baseline optimizer.
- Later Bayesian optimization backend once Python 3.10 compatibility is confirmed.

### Approach C: Visual Convergence Prototype

Build a small GUI or notebook-first prototype where a synthetic target spectrum is shown live while candidate simulations converge toward it.

Effort: M
Risk: Medium/High

Pros:

- Most visually compelling.
- Great for demos and intuition.
- Naturally leads to report generation.

Cons:

- GUI can distract from the scientific core.
- Harder to test.
- Backend and optimizer issues become UI issues too early.

Reuses:

- `xmimsim`.
- Matplotlib or Plotly.
- A simple optimizer first.

## Recommended Approach

Use Approach B, but milestone it like Approach A.

The project should get the scientific core boundaries right immediately, while keeping the first deliverable to one synthetic recovery run and one simple report. This gives the project a stable shape without delaying the first useful proof.

## Open Questions

- Which spectrum file format should be treated as the first measured-data input?
- Which geometry parameters should be optimized first: distances, angles, layer thickness, density, or a smaller subset?
- Which composition model should be optimized first: two-element mixture, layered sample, or arbitrary element fractions with normalization?
- How slow is one XMI-MSIM simulation in the target environment?
- Does the `xmimsim` wrapper work reliably in the `py10` Windows environment, or does the project need a direct CLI/XML backend adapter?
- Which uncertainty estimate is acceptable for the first milestone: local objective curvature, bootstrap/noise perturbation, repeated optimization starts, or Bayesian posterior once active learning is implemented?

## Success Criteria

Milestone one is done when:

1. autoxmimsim can generate a synthetic target spectrum from known sample parameters.
2. autoxmimsim can run candidate simulations through a backend interface.
3. autoxmimsim can compare target and candidate spectra with at least one direct whole-spectrum objective.
4. autoxmimsim can recover the synthetic truth within a documented tolerance for a small parameter space.
5. autoxmimsim writes a report containing target parameters, recovered parameters, errors, objective history, fit plot, residual plot, and a first uncertainty summary.
6. The pipeline is covered by tests using a fake or cached simulation backend so normal test runs do not require expensive Monte Carlo simulations.

## Distribution Plan

Initial distribution should stay simple:

- Python package installable from GitHub with `pip install -e .`.
- CLI entrypoint named `autoxmimsim`.
- Reports written to an output directory as HTML or Markdown plus image files.

Later distribution:

- GitHub Actions for tests on push and pull request.
- Tagged GitHub Releases once the synthetic recovery pipeline is stable.
- Optional PyPI publication after the public API and backend dependencies settle.
